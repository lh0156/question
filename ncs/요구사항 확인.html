# 현행 시스템 파악의 정의
  - 하위 시스템  확인
  - 제공 기능 확인
  - 다른 시스템들과 교환 정보 확인
  - 하드웨어를 포함한  소프트웨어를 확인
  - 네트워크의 구성 확인



# 미들웨어
- 운영체제와 소프트웨어 애플리케이션 사이에 위치하며 운영체제가 제공하는 서비스를 추가 및 확장하여 제공하는 컴퓨터 소프트웨어를 말한다

# 웹 어플리케이션 서버
- 웹상에서 클라이언트에게 웹 서비스를 제공하기 위한 어플리케이션을 동작하는 환경을 제공하는 서버이다. JBoss, GlassFish, Tomcat, Jetty, Jeus 등이 있다.


# 요구사항 개발 프로세스
- 도출 > 분석 > 명세 > 확인

# 요구사항 개발 도출
  - 소프트웨어가 해결해야 할 문제를 이해한다.
  - 요구사항 어디에 있는지 확인 한다.
  - 이해 관계자를 식별한다.
  - 개발팀과 고객사이의 관계가 만들어진다.

# 요구사항 명세서를 작성 시 고려해야할 사항
  - 체계적으로 검토한다.
  - 체계적으로 평가한다.
  - 승인될 수 있는 요구 사항을 작성한다.
  - 시스템 정의를 한다.

# 요구사항 확인 시 고려해야 할 사항
  - 분석가가 요구 사항을 이해 했는지 확인한다.
  - 요구사항의 문서가 회사의 표준에 적합하고 이해 가능 해야한다.
  - 이해 관계자들의 문서를 검토해야 한다.
  - 일관성이 있어야 한다.

# 요구사항 분석기법
  - 요구사항 분류
  - 개념 모델링
  - 요구사항 할당
  - 요구사항 협상

# 요구사항 분류의 기준
  - 요구사항이 기능인지 비기능인지를 확인한다.
  - 제품에 관한것인지 프로세스에 관한것인지를 확인한다.
  - 우선순위의 여부를 확인한다.
  - 소프트웨어 생명주기 동안에 변경이 발생하는지 여부를 판단한다.

# 비기능의 분류시 제품 분류
  - 사용성(usability)
  - 효율(efficiency)
  - 신뢰성(reliability)
  - 이식성(portability)

# 비기능의 분류 연결 관계
  - 조직 <-> 납품
  - 조직 <-> 구현
  - 조직 <-> 표준
  - 외부 <-> 상호운용성

# 분석모델의 검증방법
  - 유스케이스선택
  - 클래스도출
  - 클래스명과 속성
  - 경계 및 상세화검토

# 개념 수준의 분석 클래스 검증시 검증 항목
  - 개별 유스케이스 단위로 작성 하지 않고 시스템 전체를 대상으로 작성한다.
  - 중요도가 높은 요구사항 또는 유스케이스에 필요한 엔티티 클래스가 도출한다.
  - 도출된 클래스 이름과 설명이 이해관계자 간에 이견이 발생하지 않도록 명확한지 판단한다.
  - 클래스간의 순환적 관계가 불필요하게 정의되어 있는지 확인한다.
